主帖

Michael：为什么 OpenClaw 变成 MCP server 是一件大事

Michael2026-03-29 22:47x.com/michael_chomsky/status/2038266680296747079

这条长帖是在解释 Peter Steinberger 前一条简短更新背后的真正含义：OpenClaw 即将成为一个 MCP server。

主帖的第一层判断是：过去程序化接 OpenClaw 很痛。不是模型本身难，而是外围胶水代码很多：WebSocket、TLS、health checks、chat relay、渠道桥接，全都得自己维护。

第二层判断是：现在 OpenClaw 会直接暴露 MCP tools。像 messages_read、messages_send、events_poll、conversations_list 这样的能力，理论上可以直接被支持 MCP 的 agent 调用。

第三层判断是：架构角色被拆开了。以前 OpenClaw 同时像‘大脑’和‘消息层’；现在 Claude Code、Codex、Cursor 等都可以当大脑，而 OpenClaw 专注处理 Slack / Telegram / Discord 等渠道。

第四层判断是：真正的价值开始下沉到 messaging infra。主帖作者明确说：对云服务商来说，最难维护的，从来不是 AI 本身，而是 50+ 消息平台适配、OAuth token、bot 池、限流与基础设施稳定性。

他给出的最直观对比是：以前是‘你告诉 Claude Code → Claude Code 找 OpenClaw agent → OpenClaw agent 再发 Slack’；现在是‘你告诉 Claude Code → Claude Code 直接调用 messages_send → Slack 消息发出’。少了一跳，整个系统的可组合性和可调试性都上来了。

总结

这组帖子真正完成的三次跃迁

跃迁 1：角色变化

OpenClaw 从‘一个 agent’转向‘消息基础设施层’，更像 agent 生态的通信底座。

跃迁 2：大脑可替换

Claude Code、Codex、Cursor、cron job 等都可以成为上层 brain，而 OpenClaw 专注于消息和渠道。

跃迁 3：护城河下沉

真正难复制的不是模型，而是 50+ messaging adapters、OAuth、bot pools、限流与稳定交付。

一句话总结：这条线程真正传递的不是‘OpenClaw 新增了一个开发者接口’，而是：它正在从一个带消息能力的 agent 壳，转成 agent 生态里的通用消息底座。
