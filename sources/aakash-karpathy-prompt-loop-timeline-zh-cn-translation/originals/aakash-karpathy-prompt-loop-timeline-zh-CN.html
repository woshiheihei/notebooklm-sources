1）2026-03-20 —— 最初的框架性推文

作者： Aakash Gupta 链接： https://x.com/aakashgupta/status/2034851259442749909

现在，你只要花 25 美元、用一块 GPU，就能在一夜之间跑完 100 组实验，而且这些实验根本不需要你逐个手工设计。

Karpathy 开源了 AutoResearch。它一周内拿下 4.2 万个 GitHub star。Fortune 把它称为 “The Karpathy Loop”。

几乎所有文章都只盯着它的机器学习一面，却错过了更大的故事：这套底层模式，适用于任何可以用数字打分的东西。广告文案、冷邮件、视频脚本、招聘文案、skill 文件，都可以。

整个系统只有三个文件：一个让 agent 改；一个 agent 永远不能碰；一个由你写、用来给出优化方向。每一轮循环只要 5 分钟。分数涨了？git commit。分数跌了？git reset。一小时 12 轮，一夜 100 轮。

Karpathy 把它用在一段自己已经手工优化了数月的代码上，agent 又找出了 20 个他漏掉的改进点，让速度再快 11%。Tobi Lutke 把它指向 Shopify 的 Liquid 模板引擎，经过 93 次自动提交后，渲染性能提升了 53%。

我花了两周把这个系统拆开研究。今天这份指南会告诉你，怎样把它用到你每天真正会做的东西上：6 个使用场景、3 步搭建方法，以及那些会让实验在一开始就跑废的 eval（评测）错误。

完整指南： https://aibyaakash.com/p/autoresearch-guide

4）2026-03-28 —— 第一个明确的非 ML 例子：前端性能优化

作者： Aakash Gupta 链接： https://x.com/aakashgupta/status/2037776197719834766

有一次测试，把页面延迟从 1100ms 降到了 67ms。67 轮迭代，94% 的降幅。整个过程完全由 agent 在无人盯着的情况下完成。

问题包括：阻塞渲染的 CSS、未优化的图片加载、阻塞首次绘制的字体加载。每一个修复都不大，但叠加起来，效果极其可观。

这里的 eval 是计算机本来就会生成的数字：Lighthouse 分数、First Contentful Paint、Total Blocking Time、Cumulative Layout Shift。根本不需要人工打分，不涉及主观判断。数字下降了，就是优化成功；没降，就是没成功。

这是最容易搭起来的一种 AutoResearch 形态。如果你的 CI 流水线本来就会产出一个数字，你今晚就能把这套循环指向它：测试通过率、API 响应时间、bundle 体积、内存占用……都行。一个文件负责变化，一个指标负责优化，然后就是 git commit 或 git reset。

每个负责 Web 前端的 PM，待办列表里几乎都躺着性能优化，只是它总被更“紧急”的事情往后压。这套循环的意义在于：即使 backlog 一动不动，它也能自己跑起来。

5）2026-03-29 —— 目标推文：把这套循环用在提示词可靠性上

作者： Aakash Gupta 链接： https://x.com/aakashgupta/status/2038146639303512255

我把 Karpathy 的那套循环，应用到了每个使用 AI agent 的团队都会卡住的那个问题上：怎么把提示词从 80% 可靠 推到 95% 可靠。

这套模式几乎是原样迁移的。一个文件负责变化，一个指标负责打分。agent 每一轮只改一个地方，测试它，保留赢家，撤销输家。一小时 12 轮，一夜 100 轮。

你不再是在优化训练脚本，而是在优化任何会被反复使用的提示词或系统指令：客服 agent 的提示词、内部工作流自动化、数据抽取流水线、代码评审指令……凡是那种你曾认真写过、把它调到“差不多能用”以后，就因为手工迭代的边际收益越来越低而停下来的东西，都适用。

整个设置只需要三样东西：

1. 你想改进的目标提示词； 2. 2～3 个真实测试输入，也就是在生产环境里真的会打到这个提示词的请求； 3. 3～6 个二元的是/否检查项，用来定义“质量”。

例如：输出是否满足格式约束？是否遵循了那条特定指令？是否避开了你反复遇到的失败模式？

整个循环是这样的：

- 把提示词在所有测试输入上执行 30 次； - 用检查清单给每个输出打分； - 找出失败最频繁的那一项标准； - 只改提示词中的一个地方； - 再看分数有没有提高； - 提高了，就 git commit； - 没提高，就 git reset； - 重复，直到分数稳定超过 95%。

你第二天醒来时会看到：

- 改进后的提示词被单独保存，原始版本保持不动； - 一份 results.log，记录每一轮的分数； - 一份变更日志，说明哪些改动有效、哪些无效，以及为什么。

Karpathy 带来的、可以超越 ML 迁移出去的那个洞见是：只要你能给它打分，你就能对它做 AutoResearch。 训练损失是分数，提示词输出质量的二元检查清单也是分数。这个循环根本不关心它在优化什么，它只需要一个会上下波动的数字。

现在的提示词工程，看起来很像“还没有单元测试时代的软件开发”：手工微调、靠感觉评估、没有版本控制、没有系统化迭代。把 Karpathy Loop 用到提示词上之后，这件事就变成了一门真正的工程学科——每一轮改进都可以被衡量。

所有在跑 AI agent 的团队，手里都有一些“差不多能用”的提示词。而“差不多能用”和“足够可靠”之间的那道缝，恰恰就是这套循环能在你睡觉时替你补上的。

6）2026-03-29 —— 后续补充：把这套架构映射关系讲透

作者： Aakash Gupta 链接： https://x.com/aakashgupta/status/2038132294817656978

AutoResearch 之所以能在一周内拿下 4.2 万个 GitHub star，是因为这套架构能迁移到任何有分数的对象上。

Karpathy 最初是把它用在机器学习训练上：train.py 是 agent 会改的代码；val_bpb 是度量指标；program.md 是人类提供的研究方向；prepare.py 是锁死的 eval harness（评测护栏）。分数变好就 git commit，变差就 git reset。

我把它迁移到提示词工程上时，整个映射大概只花了十分钟，因为每个组件都有直接对应物：

- train.py → 你的 skill 文件或 system prompt 文件； - val_bpb → 一个通过/失败检查清单，也就是对每个输出打 3～6 个是/否分； - program.md → 你写给 agent 的说明，告诉它该优化什么、必须遵守哪些约束； - prepare.py → 一个锁死的 eval 脚本，agent 可以先构建它，但之后永远不能再修改它； - Git 的工作方式完全不变。

之所以这套架构稳得住，是因为 Karpathy 做了一个几乎没人真正拿出来讨论的设计决定：他把系统清晰地拆成了四个角色——

1. 一个会变化的文件； 2. 一个负责裁判的指标； 3. 一个负责给方向的说明； 4. 一个负责上锁的约束。

这四个角色，几乎描述了所有有效优化循环的共同结构：A/B 测试、临床试验、精益制造、PDCA，甚至科学方法本身。

大多数 AI agent 框架失败，就是因为它们把这些边界混在了一起：写代码的 agent 同时也在评估代码；负责设定目标的系统同时也在衡量自己是否接近目标。AutoResearch 之所以有效，恰恰是因为负责变异文件的 agent，对“这个变异是怎么被打分的”没有任何控制权。

把它迁到提示词工程之后，产出的结果和 Karpathy 在 ML 里得到的东西是一样的：

- 一个单独保存的改进版本，原始文件保持不动； - 一份记录每轮分数的结果日志； - 一份变更日志，解释 agent 试了什么、什么有效、什么无效。

大约一小时 12 轮，一夜 100 轮，算力成本大约 25 美元。

大多数人最容易跳过、但也最关键的部分，就是那个锁死的 eval。没有它，agent 优化的就不是提示词，而是测试本身。

只要你能为“什么叫好”定义出 3～6 个二元标准，你就可以把这套循环用到任何东西上：提示词、邮件序列、落地页文案、onboarding 流程、客服脚本。Karpathy Loop 本质上是一种伪装成 ML 工具的通用优化架构。

时间线压缩总结

把这些帖子连起来看，这条时间线非常清晰：

  • 3 月 20 日 —— AutoResearch 被重新定义为一种普适的优化模式，而不只是 ML 小技巧。
  • 3 月 27～28 日 —— 外部讨论开始放大这个判断：越来越多人意识到，大家严重低估了它；Aakash 也开始把它推向 PM 与非 ML 场景。
  • 3 月 28 日 —— 第一个明确的非 ML 用例出现：用机器自动生成的指标去做前端性能优化。
  • 3 月 29 日 —— 最关键的一跳发生了：这套循环被完整迁移到提示词工程和 AI agent 可靠性问题上。
  • 3 月 29 日（跟进帖） —— 这套方法被进一步形式化：可变文件、评分指标、指导方向、锁死约束，构成了它的最小通用架构。

换句话说，这条时间线真正完成的，不只是“介绍一个工具”，而是把 Karpathy Loop 从一个看起来属于机器学习研究者的工作流，升级成了一种可以作用于任何反复执行的指令系统的通用方法论。

一句话提炼

这组推文真正推动的，不是一个更聪明的 agent，而是一种新的工作范式：把原本靠感觉手调的提示词、流程和脚本，变成可以被锁定评测、批量实验、自动迭代的工程对象。
